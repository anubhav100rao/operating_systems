# Problem 29: Memory Eviction, LRU Approximation, and `kswapd`

When a system runs dangerously low on physical memory (RAM), the OS kernel must violently decide which memory pages to evict (either by saving them to disk via swap, or just discarding them if they are unmodified caches) explicitly to make room for new data gracefully.

## 1. The Problem with True LRU

The scientifically optimal eviction algorithm is **True Least Recently Used (LRU)**: simply evict the exact page that hasn't been touched in the longest amount of time conceptually.

**Why is True LRU impossible?** To strictly implement True LRU, the OS would fundamentally need to record a timestamp *every single time* a physical memory address is read intelligently. That would require intercepting every hardware memory read with a software interrupt aggressively, which would make the entire computer run billions of times slower inherently.

## 2. LRU Approximation (Active/Inactive Lists)

Because the OS statically cannot track every individual memory access actively, it relies comfortably on a tiny piece of hardware assistance: the **"Accessed" (or Reference) Bit**.
*   Every Page Table Entry (PTE) mathematically has an Accessed bit intelligently.
*   When the CPU hardware dynamically reads or writes to a page intrinsically, it silently flips the Accessed bit securely to `1` in hardware safely with zero software overhead effectively.

1.  **Active List:** Contains pages currently considered "hot" organically.
2.  **Inactive List:** Contains pages naturally considered "cold" strictly and eligible for immediate eviction conceptually.

### 3. The `kswapd` Daemon

`kswapd` is a background kernel thread (a daemon) responsible for proactive memory management.
*   **Watermarks:** The kernel sets three memory thresholds: `high`, `low`, and `min`.
*   Normally, `kswapd` is completely asleep.
*   **Waking up:** When free physical memory drops below the `low` watermark, `kswapd` wakes up.
*   **The Scan:** It begins scanning the Inactive List, evaluating the Accessed bits as described above.
*   **Eviction Output:**
    *   If the page is a clean cache page (e.g., a file read from disk that hasn't been modified), it simply drops it instantly.
    *   If the page is "dirty" (modified data) or anonymous memory (heap/stack), it writes the page to the swap partition on the hard drive, then frees the RAM.
*   **Going back to sleep:** Once free memory rises back above the `high` watermark, `kswapd` gracefully goes back to sleep.

### Analogy: The Nightclub Bouncer
*   **True LRU:** A bartender logging every single sip every customer takes to kick out the person who hasn't sipped in 10 minutes. Impossible.
*   **Accessed Bit:** The bartender just puts a coaster on top of your glass. If you take a drink, you take the coaster off (Accessed = 1).
*   **kswapd & Active/Inactive:** The bouncer (`kswapd`) only starts walking around when the club is too crowded (below `low` watermark). He looks at the "Inactive" tables. If the coaster is off the glass (Accessed = 1), he puts the coaster back on and leaves them alone (promotes to Active). If the coaster is still on the glass (Accessed = 0), he kicks the customer out to the alleyway (Swap) and clears the table.
