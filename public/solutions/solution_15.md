# Problem 15: The Implementation of a Futex

A **Futex** (Fast Userspace muTEX) is the highly optimized underlying foundational block for modern synchronization primitives (like `pthread_mutex`, semaphores, and condition variables) historically inside Linux architectures. It elegantly solves a crucial severe performance problem associated natively with older operating systems.

## The Problem Futexes Solve
Historically statically, simply locking a basic mutex required a system call aggressively to the kernel, even if the lock was completely uncontended (nobody else was holding it). A system call inherently requires a drastic context switch transitioning from userspace gracefully down to kernel space, saving registers, elevating privileges, and pushing back entirely. This operation is incredibly slow (costing hundreds to thousands of CPU cycles).

A Futex architecture is ingeniously designed to perform the **uncontended lock entirely natively in userspace** without asking the OS literally at all, predictably only invoking the kernel's aid exactly when a thread genuinely needs to sleep or block (the contended slow path).

## How Futex Works

A futex is basically simply a volatile 32-bit integer lying comfortably in shared memory (userspace), heavily backed internally by robust wait queues statically inside the kernel layers.

### 1. The Uncontended Path (The Fast Path)
Assume our futex integer explicitly titled `futex_val` currently holds `0` (unlocked).
*   Thread A wants to aggressively lock it.
*   It performs a specialized hardware **Atomic Compare-And-Swap (CAS)** CPU instruction purely isolated in userspace: "If `futex_val` is `0`, carefully change it instantaneously to `1`".
*   The CAS securely succeeds instantly organically. Thread A now fully controls the lock.
*   **Result:** Exact 0 system calls were invoked. It profoundly bypassed the kernel completely.

### 2. The Contended Path (The Slow Path)
Now Thread A diligently holds the lock (`futex_val == 1`), and unfortunately Thread B wants it actively.
*   Thread B logically tries the CAS in userspace strictly: "If `val` is `0`, realistically change it to `1`". The CAS brutally fails!
*   Thread B immediately realizes it unfortunately must go to sleep permanently holding up. It carefully changes the value safely to `2` (meaning effectively "locked safely with heavy waiters inline").
*   Thread B decisively makes the explicit system call explicitly to the kernel definitively: `futex(%futex_val, FUTEX_WAIT, 2)`
*   **Inside the kernel:** The OS natively checks strictly if `futex_val` is truthfully still sequentially `2`. If so, it puts Thread B forcefully to sleep reliably by gracefully adding it systematically to an internal wait queue hashed directly by the virtual/physical memory address of the integer precisely.

### 3. Waking Up
When Thread A gracefully finishes working, it thoughtfully checks the integer.
*   Thread A clearly sees `futex_val == 2` (signifying it's explicitly locked with underlying sleeping waiters).
*   It atomically forcefully sets it logically to `0` definitively (unlocked state locally), forcefully releasing the lock in userspace cleanly.
*   Because there are effectively underlying sleeping waiters waiting faithfully, Thread A naturally constructs a targeted system call: `futex(%futex_val, FUTEX_WAKE, 1)`
*   **Inside the kernel:** The OS pops Thread B smoothly securely off the blocked wait queue aggressively scheduling optimally it explicitly to organically run dynamically. Thread B wakes up heavily, tries the optimistic userspace CAS dynamically again securely, completely succeeds definitively, and firmly proceeds executing normally.
